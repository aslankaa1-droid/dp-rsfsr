# Программа и методика испытаний

## ЦП РСФСР — мобильное приложение

**Условное обозначение:** РСФСР.МП.10.ПМИ
**Стандарт:** ГОСТ 19.301-79

---

## 1. Объект испытаний

### 1.1. Наименование
Мобильное приложение «ЦП РСФСР — мобильное приложение залогодателя» во всех целевых сборках:
- Android (RuStore, Google Play);
- iOS (App Store);
- Windows (Microsoft Store);
- PWA (любой современный браузер).

### 1.2. Цель испытаний
Подтверждение соответствия программы требованиям технического задания (РСФСР.МП.01.ТЗ), нормативным требованиям регуляторов и стандартам качества программных продуктов.

## 2. Виды испытаний

### 2.1. Функциональные испытания
Проверка соответствия функциональным требованиям ТЗ. Объект — все 23 экрана и все переходы.

### 2.2. Нагрузочные испытания
Проверка производительности при пиковых нагрузках.

### 2.3. Испытания безопасности
Сертификационные испытания по требованиям ФСТЭК России, ФСБ России.

### 2.4. Испытания доступности
Проверка соответствия WCAG 2.2 уровня AA.

### 2.5. Испытания совместимости
Проверка работы на различных моделях устройств и версиях ОС.

### 2.6. Регрессионные испытания
Автоматические тесты на каждом релизе.

### 2.3. Сертификационные испытания криптографии и защиты (MASVS L2+)

Введены в исполнение требований OWASP MASVS v2.0 уровня L2 (для приложений в финансовой инфраструктуре) и нормативов ФСБ России по применению СКЗИ.

| ID | Сценарий | Описание | Ожидаемый результат | MASVS-привязка |
|---|---|---|---|---|
| FT-021 | Двухуровневый certificate pinning (MITM-атака) | На стороне MITM-прокси подменяется сертификат API-домена. Тест выполняется поочерёдно для primary-пина и backup-пина. | Запросы отклоняются с ошибкой `SSL_PINNING_VIOLATION`; событие пишется в локальный security-log и асинхронно — в SIEM оператора. | MASVS-NETWORK-1, L2 |
| FT-022 | Root / Jailbreak detection | На Android используется RootBeer (5 проверок: su-binary, dangerous apps, busybox, build tags, RW system); на iOS — IOSSecuritySuite (jailbroken file paths, suspicious symlinks, can write outside sandbox, fork() returns success, dynamic linker hooks). | При обнаружении root/jailbreak — блокировка доступа к criticality-функциям (выпуск ЦФА, подпись смарт-контракта), пользователю показывается экран с объяснением и контактом поддержки. | MASVS-RESILIENCE-1, L2 |
| FT-023 | ГОСТ-канал payload (Magma / Kuznyechik tunnel) | Все вызовы к API-шлюзу оборачиваются в ГОСТ TLS-туннель (RFC 9189) с шифрсьютами по ГОСТ Р 34.12-2015. Тестируется handshake, обмен сообщениями, ротация сессионных ключей. | Handshake завершается успешно; сессионные ключи ротируются каждые 60 минут / 1 ГБ трафика; payload не расшифровывается без действующего сессионного ключа. | MASVS-NETWORK-2, L2 |
| FT-024 | Secure Enclave / Strongbox для приватных ключей | Приватный ключ УКЭП хранится в Secure Enclave (iOS) / Strongbox-backed KeyStore (Android API ≥28). Тест: попытка экспорта приватного ключа через debug-канал, root-эксплойт, физический доступ. | Ключ не покидает аппаратный модуль; попытки экспорта возвращают `KEY_EXPORT_DENIED`; биометрический prompt требуется при каждой подписи. | MASVS-CRYPTO-1, MASVS-PLATFORM-2, L2 |
| FT-025 | Верификация подписи СКЗИ | На стороне приложения проверяется подпись ответов API-шлюза по ГОСТ Р 34.10-2012 (длина ключа 256/512). Тест: валидная подпись, подменённая подпись, истёкший сертификат. | Валидная — ответ принимается; подменённая — `INVALID_SIGNATURE` + блокировка; истёкший сертификат — `CERT_EXPIRED` + триггер обновления цепочки. | MASVS-CRYPTO-2, L2 |

Покрытие FT-021…FT-025 обязательно для прохождения добровольной сертификации в системе сертификации СКЗИ ФСБ России и для подачи в реестр операторов ИС ЦФА (779-П / Указание 4336-У).

### 3.2. Этап 2. Бета-тестирование (закрытое)

Закрытый круг пользователей до 1 000 человек, в течение 30 дней. Метрики:
- Количество завершённых пользовательских сценариев.
- Среднее время прохождения каждого сценария.
- Количество ошибок на 100 операций.
- Crash-free sessions (целевое: ≥ 99,8%).

### 3.3. Этап 3. Приёмочные испытания

С участием заказчика, регулятора (Банк России), представителей профильных министерств. Проверка соответствия всем требованиям ТЗ.

### 3.4. Этап 4. Сертификационные испытания

#### Сертификация СЗИ
- Аттестация по требованиям приказа ФСТЭК России №239 (КИИ).

- Соответствие ГОСТ Р 57580.1 уровня 4.
- Сертификация криптографических средств ФСБ России.

#### Сертификация ПО
- Включение в Единый реестр российского ПО (ПП РФ №1236).

## 4. Методика испытаний

### 4.1. Функциональные испытания

#### Подготовка
1. Создаётся тестовое окружение с моками внешних систем (ЕСИА, ЕБС, ЕГРН, ЦБ).
2. Подготавливается набор тестовых пользователей с разными KYC-статусами.
3. Подготавливаются тестовые активы и ЦФА в различных состояниях.

#### Проведение
- Автоматические тесты через Cypress / Playwright для PWA-режима.
- Capacitor Test Kit (Appium-обвязка над нативным wrapper'ом) для iOS/Android-сборок.
- WebView2 Test Harness для Windows-сборки.
- XCUITest для iOS native-расширений (биометрия, Secure Enclave, ГОСТ-канал).
- Espresso для Android native-расширений (Strongbox, BiometricPrompt, СКЗИ).

#### Критерии успеха
- 100% покрытие сценариев FT-001 — FT-020 (функциональные).
- 100% покрытие сценариев FT-021 — FT-025 (криптография и защита, MASVS L2+).
- 0 критических ошибок.
- Не более 5 косметических замечаний.

### 4.2. Нагрузочные испытания

#### Подготовка
- Тестовая среда, аналогичная продакшну.
- JMeter / Yandex Tank для серверной нагрузки.
- Lighthouse / WebPageTest для клиентских метрик.

#### Сценарии
- 5 000 RPS на API-шлюзе в течение 30 минут.
- 50 000 одновременных пользователей.
- Регистрация 1 000 новых пользователей за 5 минут.
- Выпуск 100 ЦФА за час.

#### Критерии успеха
- Время отклика P95 ≤ 200 мс.
- Error rate ≤ 0,1%.
- Cold start ≤ 2 секунды на 4G.
- Memory peak ≤ 300 МБ.

### 4.3. Испытания безопасности

#### Виды проверок
- Статический анализ кода (SAST): PT Application Inspector, PVS-Studio.
- Динамический анализ (DAST): PT BlackBox.
- Анализ компонентов (SCA): обнаружение уязвимых библиотек.
- Внешний пентест аккредитованной лабораторией.
- Внутренний Red Team.
- Проверка на соответствие OWASP MASVS L2+.

#### Критерии успеха
- 0 уязвимостей категорий Critical и High.
- Не более 5 уязвимостей Medium с принятым риском.
- Подтверждение соответствия ГОСТ Р 57580.1 уровня 4.

### 4.4. Испытания доступности

#### Виды проверок
- Автоматическая проверка через axe-core / WAVE.
- Тестирование с экранными читалками (VoiceOver, TalkBack, Narrator).
- Тестирование с увеличенным шрифтом.
- Проверка контрастности (≥ 4,5:1 для основного текста).

#### Критерии успеха
- Соответствие WCAG 2.2 уровня AA на 100%.
- Все интерактивные элементы доступны с клавиатуры.
- Все изображения имеют alt-атрибуты.
