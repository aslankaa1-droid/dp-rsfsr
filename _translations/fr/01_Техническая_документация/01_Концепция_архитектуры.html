Concept d’architecture de CPFSR

Plateforme Numérique « Service Russe de Financement du Développement Social »

Auteur et titulaire des droits du concept : Abdul-Khakim Akhmadovitch Kagirov. Certificat de dépôt : N° 4011265 du 19.12.2024 (ireg.pro), empreinte GOST R 34.11-2012. Le document est conforme aux exigences de GOST 34.602 (Cahier des charges pour la création d’un système automatisé), GOST 19.701 (schémas d’algorithmes), GOST R 57580.1.

1. Objet et domaine d’application

La plateforme CPFSR est un système d’information opérateur d’importance étatique pour l’émission d’actifs financiers numériques (AFN) garantis par les biens des citoyens et des organisations de la Fédération de Russie. En régime pilote-light de 2026–2028, le financement ciblé des projets nationaux et d’infrastructure contre la garantie d’un pool d’AFN est organisé via le circuit d’infrastructure de VEB.RF (Usine de financement de projets, décret du Gouvernement de la Fédération de Russie N° 158 du 15.02.2018) et le Trésor fédéral dans le cadre du Code budgétaire en vigueur de la Fédération de Russie. La Banque de Russie agit en tant que régulateur de supervision général du marché financier au titre de l’art. 4 de la Loi fédérale N° 86-FZ. À l’horizon 2029+, une transition vers un modèle d’émission garantie ciblée par la Banque de Russie est possible (avec l’utilisation du rouble numérique) sous réserve de l’adoption du paquet correspondant d’actes normatifs et d’un track-record positif du pilote-light.

Le document décrit : - les principes architecturaux généraux de la plateforme ; - l’architecture logique et physique ; - la pile technologique sur des composants entièrement substitués par les importations ; - le modèle de fiabilité et de tolérance aux pannes ; - les exigences relatives à la classification du système d’information en tant qu’objet d’infrastructure d’information critique (IIC).

3. Architecture logique — 7 couches

  1. Canal du propriétaire de l’actif — portail web, applications mobiles iOS/Android/Windows, chatbot dans les canaux d’État.
  2. Backend-for-Frontend (BFF) — une couche d’agrégation optimisée pour chaque canal.
  3. API Gateway — point d’entrée unique, authentification, rate-limiting, filtres antifraude.
  4. Services métier — microservices : KYC, évaluation, assurance, émission d’AFN, contrats intelligents, registre, adaptateur escrow, reporting, notifications.
  5. Couche de données — SGBD relationnels (Postgres Pro / Red DB), stockages de documents, cache in-memory (un analogue national de Redis), stockage objet compatible S3.
  6. Couche d’intégrations — adaptateurs vers les systèmes d’information d’État (Gosuslugi, EGRN, SMEV 3, GIS GMP, FK, Banque de Russie, Rosfinmonitoring, FNS, NSPK).
  7. Couche de sécurité — DLP, SIEM, EDR, agent GosSOPKA, modules HSM (KriptoPro HSM), systèmes de surveillance et d’audit.

4. Architecture physique

4.1. Géographie

  • Au moins 3 centres de données opérationnels sur le territoire de la Fédération de Russie dans différents districts fédéraux (Central, Sibérien, de l’Oural — un exemple).
  • Topologie active-active-active avec réplication synchrone pour les circuits critiques.
  • Centre de données « froid » de secours pour les scénarios de reprise après sinistre.

4.2. Modèle réseau

  • Protection à deux niveaux du périmètre : niveau 1 — le réseau d’État « RTKomm », niveau 2 — propre périmètre avec NGFW nationaux.
  • Segment isolé pour les intégrations avec la Banque de Russie (SPFS, canal protégé ViPNet).
  • Segmentation complète par microservices (réseau zero-trust).

4.3. Plateforme d’exécution

  • Conteneurisation sur des distributions nationales de Kubernetes (Deckhouse Stronghold / analogues Yandex Managed K8s on-prem).
  • Service Mesh avec des certificats KriptoPro pour mTLS.

5. Pile technologique

5.1. Logiciels système

  • Systèmes d’exploitation : Astra Linux Special Edition, Red OS, Alt Server — certifiés FSTEC.
  • Hyperviseur : zVirt / SberCloud / vStack (analogues nationaux de VMware).
  • Conteneurisation : distributions nationales de Kubernetes.

5.2. Bases de données et stockages

  • Relationnelles : Postgres Pro Enterprise, Tantor Special Edition, Red DB.
  • Analytiques : Pangolin, Arenadata DB.
  • Orientées document : Postgres Pro Multimaster JSONB ; ou un analogue national de MongoDB (Atlasync).
  • Objet : Yandex Object Storage on-prem, Arkuda, hôte compatible S3.

5.3. Brokers et traitement de flux

  • Apache Kafka dans les distributions Arenadata Streaming / Yandex Managed Service for Apache Kafka.
  • RabbitMQ comme fallback pour les files d’attente classiques.

5.4. Couche DLT (registre distribué d’AFN)

  • Basée sur une plateforme nationale de la classe « Masterchain » / Waves Enterprise ; le fournisseur spécifique est convenu avec la Banque de Russie à l’étape pilote.
  • Hachage des opérations — GOST R 34.11-2012.
  • Signature électronique des opérations du registre — GOST R 34.10-2012.

5.5. Cryptographie

  • HSM : KriptoPro HSM / ViPNet HSM.
  • SDK : KriptoPro CSP 5.0, ViPNet CSP.
  • Certificats : certificats qualifiés des autorités de certification accréditées par le Ministère du Développement numérique.

5.6. Sécurité de l’information

  • SIEM : KOMRAD, MaxPatrol SIEM, R-Vision SIEM.
  • EDR : Kaspersky EDR / R-Vision EDR.
  • DLP : Solar Dozor / InfoWatch.
  • WAF : PT Application Firewall / SolidWall.
  • Systèmes d’analyse de code : PT Application Inspector, AppSec.Hub.
  • VPN : ViPNet, KontinentAP.

5.7. Observabilité

  • Métriques : VictoriaMetrics, une pile Prometheus nationale.
  • Journaux : Loki / un analogue national.
  • Traçage : Jaeger / une pile OpenTelemetry nationale.
  • Tableaux de bord : Grafana OSS sur sa propre infrastructure.

6. Principes architecturaux

  1. L’importance étatique est primordiale. La sécurité, la fiabilité et la compatibilité réglementaire priment sur la vitesse de développement.
  2. Indépendance à 100 % vis-à-vis des importations. Tous les composants — du registre du Ministère du Développement numérique ou certifiés par le FSTEC.
  3. Event-driven. Tous les processus métier (émission d’AFN, échange, événement d’assurance) sont des événements asynchrones avec traçabilité.
  4. Journal d’audit immuable. Toutes les opérations clés sont écrites dans le registre DLT et dans un journal d’audit centralisé ; les données sont immuables.
  5. Zero-Trust. Chaque appel entre microservices est effectué avec authentification mTLS.
  6. Secure by Design. La sécurité est intégrée dès l’étape de l’architecture (Privacy by Design, KIIS by Design).
  7. Cloud-agnostique on-prem. La plateforme est déployée sur sa propre infrastructure sans dépendance vis-à-vis des clouds étrangers.
  8. API-first. Toutes les intégrations passent par des API documentées (OpenAPI 3.1).
  9. GitOps + IaC. L’infrastructure et les configurations sont en git avec déploiement automatique (Argo CD / un analogue national).
  10. Niveau de redondance N+2. Les services critiques disposent d’une double redondance.
