# Exigences non fonctionnelles (NFR) de CPFSR

Le document détaille les exigences en matière de performance, de fiabilité, de sécurité de l'information, d'observabilité et d'exploitation de la plateforme.

---

## 1. Performance

| Indicateur | Valeur cible | Méthode de mesure |
|---|---|---|
| Charge de pointe en entrée | 5 000 RPS sur la passerelle API | Tests de charge (JMeter / Yandex Tank) |
| Temps de réponse moyen de l'API métier | ≤ 200 ms (P95) | Prometheus / OpenTelemetry |
| Temps de réponse des opérations « lourdes » (émission d'AFN) | ≤ 30 sec E2E | Synthetic monitoring |
| Temps d'affichage du premier écran de l'application mobile | ≤ 1,5 sec en 4G | Real User Monitoring |
| TPS du registre DLT | ≥ 200 transactions/sec | Profilage du nœud DLT |
| Utilisateurs concurrents | ≥ 50 000 simultanés | Tests de charge stressante |

## 2. Fiabilité et disponibilité

| Indicateur | Valeur cible |
|---|---|
| SLA annuel | 99,99 % (≤ 52 min d'indisponibilité/an) |
| MTTR (temps moyen de réparation) | ≤ 15 min pour les circuits critiques |
| MTBF (temps moyen entre pannes) | ≥ 8 760 h pour les composants critiques |
| RPO (point de récupération) | 0 pour le registre DLT et les données AFN ; ≤ 5 min pour le reste |
| RTO (temps de récupération) | ≤ 5 min pour critique, ≤ 30 min pour le reste |
| Géo-distribution | 3 centres de données actif-actif |
| Redondance de la BD | Réplication synchrone ≥ N+2 |
| Backup | Incrémentiel horaire + complet quotidien, rétention 7 ans |

## 3. Scalabilité

1. **Mise à l'échelle horizontale** de tous les services sans état via Kubernetes HPA.
2. **Mise à l'échelle verticale** des composants avec état (BD) — jusqu'à 1 Po de stockage par cluster.
3. **Sharding** du registre AFN par identifiant du débiteur sur gage pour les grands volumes.
4. **Modèle de distribution partenaire** pour la charge des modules d'évaluation et d'assurance.

## 4. Sécurité de l'information

### 4.1. Niveau de protection
La plateforme est conforme au **4ᵉ niveau (maximum) de protection au titre de GOST R 57580.1** et met en œuvre l'ensemble des 8 domaines de protection :
- OZD — protection des données ;
- OPI — protection du périmètre du SI ;
- RSI — contrôle et délimitation des accès ;
- MAU — surveillance, audit, réponse ;
- OPT — protection continue ;
- UPI — gestion des incidents ;
- OZU — protection des comptes ;
- RTsK — réponse aux attaques informatiques.

### 4.2. Cryptographie
- Uniquement des moyens certifiés : KriptoPro CSP 5.0, ViPNet CSP, KriptoPro HSM.
- Algorithmes : GOST R 34.10-2012 (signature), GOST R 34.11-2012 (hachage), GOST R 34.12-2015 / 34.13-2015 (chiffrement).
- Canaux TLS — au moyen de certificats GOST (KriptoPro TLS).

### 4.3. Protection des données personnelles
- Conformité à FZ-152 et à l'arrêté FSTEC N° 21.
- Niveau de sécurité ISPDn — **UZ-1** (selon le décret du Gouvernement N° 1119).
- Inscription au registre des opérateurs de données personnelles de Roskomnadzor.
- Toutes les opérations sur les données personnelles sont journalisées dans le journal d'audit immuable.

### 4.4. Protection IIC
- Catégorie de signification — **1ʳᵉ (la plus élevée)** (FZ-187).
- Raccordement à GosSOPKA.
- Conformité à l'arrêté FSTEC N° 239.
- Interaction continue avec le NCCCI.

## 5. Observabilité

### 5.1. Métriques
- Métriques métier : nombre d'AFN émis, volume d'émission, temps d'émission, conversion.
- Métriques techniques : RPS, latence, taux d'erreurs, saturation des ressources.
- Métriques SI : nombre d'incidents, requêtes bloquées, autorisations échouées.
- Rétention des métriques — 13 mois dans VictoriaMetrics avec agrégation par périodes.

### 5.2. Journaux
- Journaux JSON structurés pour tous les services.
- Collecte centralisée dans Loki / un analogue national.
- Corrélation via trace_id.
- Rétention des journaux opérationnels — 13 mois.
- Rétention des journaux d'audit — 10 ans (en cohérence avec FZ-152 et le Code des impôts de la FR).

### 5.3. Traçage
- Pile OpenTelemetry pour tous les services.
- Échantillonnage à 10 % pour les requêtes standard, 100 % pour les circuits critiques (émission, échange d'AFN).
- Rétention des traces — 30 jours (HOT), 180 jours (COLD).

### 5.4. Alerting
- Système à deux niveaux : alertes automatiques + équipe de garde SOC.
- Escalade : L1 (1 min) → L2 (5 min) → L3 (15 min) → direction de l'OO (30 min).

## 6. Exploitation

### 6.1. CI/CD
- Dépôt Git — GitFlic / GitVerse / GitLab on-prem propriétaire.
- CI : pipelines avec étapes obligatoires :
  1. Analyse statique du code (PT Application Inspector, PVS-Studio).
  2. SAST / DAST (PT AppSec).
  3. Analyse de composition (détection de bibliothèques vulnérables).
  4. Tests unitaires (couverture ≥ 80 %).
  5. Tests d'intégration.
  6. Smoke-tests de performance.
  7. Signature UKEP de la build de release.
- CD : GitOps (Argo CD / un analogue national), automatisation complète des déploiements, avec mécanisme de déploiement canari.

### 6.2. Gestion de configuration
- Toutes les configurations — dans Git (Infrastructure as Code, Configuration as Code).
- Secrets — via HashiCorp Vault (ou un analogue national) avec chiffrement sur HSM.

### 6.3. Backup et DR
- Tests complets de scénarios DR — trimestriels.
- Recovery drill — une fois par semestre avec la participation du SOC.

## 7. Compatibilité et standardisation

- API : REST (OpenAPI 3.1) pour les intégrations externes ; gRPC entre services internes.
- Documentation API — génération automatique en style Swagger.
- Prise en charge des normes : OAuth 2.1, OpenID Connect, mTLS, JWT avec RS256/GostR3410.
- Conformité à ISO/IEC 25010 (attributs de qualité logicielle).

## 8. Exigences relatives aux équipes et à l'exploitation

- NOC (Network Operations Center) 24×7.
- SOC (Security Operations Center) 24×7 avec intégration à GosSOPKA.
- Équipes de garde : ≥ 6 personnes par shift pour les circuits critiques.
- Pratiques régulières Red Team / Blue Team — tous les 6 mois.
- Pentest externe annuel par un laboratoire accrédité par le FSTEC.

## 9. Exigences en matière d'indépendance vis-à-vis des importations

100 % des composants matériels et logiciels doivent :
- être inscrits au Registre unique des logiciels russes (décret du Gouvernement N° 1236), ou
- être certifiés par le FSTEC de Russie / le FSB de Russie, ou
- être un développement propriétaire de l'opérateur avec code source ouvert et assemblage sur le territoire de la FR.
