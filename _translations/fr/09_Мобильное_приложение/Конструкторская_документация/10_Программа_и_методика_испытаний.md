# Programme et méthodologie d'essais

## CPFSR — application mobile

**Identifiant :** РСФСР.МП.10.ПМИ
**Norme :** GOST 19.301-79

---

## 1. Objet des essais

### 1.1. Nom
« CPFSR — application mobile du débiteur sur gage » dans toutes les builds cibles :
- Android (RuStore, Google Play) ;
- iOS (App Store) ;
- Windows (Microsoft Store) ;
- PWA (tout navigateur moderne).

### 1.2. Objectif des essais
Confirmation de la conformité du programme aux exigences du cahier des charges (РСФСР.МП.01.ТЗ), aux exigences normatives des régulateurs et aux standards de qualité des produits logiciels.

## 2. Types d'essais

### 2.1. Essais fonctionnels
Vérification de la conformité aux exigences fonctionnelles du CdC. Objet — l'ensemble des 23 écrans et de toutes les transitions.

### 2.2. Essais de charge
Vérification de la performance sous charges de pointe.

### 2.3. Essais de sécurité
Essais de certification au titre des exigences du FSTEC de Russie, du FSB de Russie.

### 2.4. Essais d'accessibilité
Vérification de la conformité à WCAG 2.2 niveau AA.

### 2.5. Essais de compatibilité
Vérification du fonctionnement sur différents modèles d'appareils et versions d'OS.

### 2.6. Essais de régression
Tests automatiques à chaque release.

### 2.3. Essais de certification de la cryptographie et de la protection (MASVS L2+)

Introduits pour mettre en œuvre les exigences d'OWASP MASVS v2.0 niveau L2 (pour les applications de l'infrastructure financière) et les règlements du FSB de Russie sur l'usage des SZI.

| ID | Scénario | Description | Résultat attendu | Lien MASVS |
|---|---|---|---|---|
| FT-021 | Certificate pinning à deux niveaux (attaque MITM) | Côté proxy MITM, le certificat du domaine API est substitué. Le test est exécuté tour à tour pour le pin primaire et le pin de secours. | Les requêtes sont rejetées avec l'erreur `SSL_PINNING_VIOLATION` ; l'événement est consigné dans le security-log local et de manière asynchrone — dans le SIEM de l'opérateur. | MASVS-NETWORK-1, L2 |
| FT-022 | Détection Root / Jailbreak | Sous Android, RootBeer est utilisé (5 vérifications : su-binary, dangerous apps, busybox, build tags, RW system) ; sous iOS — IOSSecuritySuite (chemins de fichiers jailbroken, symlinks suspects, écriture hors sandbox, fork() retourne success, hooks dynamic linker). | En cas de détection root/jailbreak — blocage de l'accès aux fonctions critiques (émission d'AFN, signature de contrat intelligent), un écran s'affiche avec explication et contact du support. | MASVS-RESILIENCE-1, L2 |
| FT-023 | Charge utile en canal GOST (tunnel Magma / Kuznyechik) | Tous les appels vers la passerelle API sont enveloppés dans un tunnel GOST TLS (RFC 9189) avec ciphersuites au titre de GOST R 34.12-2015. Le handshake, l'échange de messages et la rotation des clés de session sont testés. | Le handshake se termine avec succès ; les clés de session sont tournées toutes les 60 minutes / 1 Go de trafic ; la charge utile n'est pas déchiffrée sans clé de session valide. | MASVS-NETWORK-2, L2 |
| FT-024 | Secure Enclave / Strongbox pour les clés privées | La clé privée UKEP est stockée dans Secure Enclave (iOS) / KeyStore adossé à Strongbox (Android API ≥28). Test : tentative d'export de la clé privée via canal debug, exploit root, accès physique. | La clé ne quitte pas le module matériel ; les tentatives d'export retournent `KEY_EXPORT_DENIED` ; le prompt biométrique est requis à chaque signature. | MASVS-CRYPTO-1, MASVS-PLATFORM-2, L2 |
| FT-025 | Vérification de la signature SZI | Côté application, la signature des réponses de la passerelle API est vérifiée au titre de GOST R 34.10-2012 (longueur de clé 256/512). Test : signature valide, signature substituée, certificat expiré. | Valide — la réponse est acceptée ; substituée — `INVALID_SIGNATURE` + blocage ; certificat expiré — `CERT_EXPIRED` + déclenchement de la mise à jour de la chaîne. | MASVS-CRYPTO-2, L2 |

La couverture FT-021 … FT-025 est obligatoire pour le passage de la certification volontaire dans le système de certification SZI du FSB de Russie et pour la soumission au registre des opérateurs SI AFN (779-P / Instruction 4336-У).

### 3.2. Étape 2. Tests bêta (fermés)

Cercle fermé d'utilisateurs jusqu'à 1 000 personnes, sur 30 jours. Métriques :
- Nombre de scénarios utilisateurs achevés.
- Temps moyen de passage de chaque scénario.
- Nombre d'erreurs pour 100 opérations.
- Sessions sans crash (cible : ≥ 99,8 %).

### 3.3. Étape 3. Essais de réception

Avec la participation du client, du régulateur (Banque de Russie), des représentants des ministères concernés. Vérification de la conformité à toutes les exigences du CdC.

### 3.4. Étape 4. Essais de certification

#### Certification SZI
- Attestation au titre de l'arrêté FSTEC N° 239 (IIC).
- Attestation au titre de l'arrêté FSTEC N° 21 (DCP, UZ-1).
- Conformité à GOST R 57580.1 niveau 4.
- Certification des moyens cryptographiques par le FSB de Russie.

#### Certification du logiciel
- Inscription au Registre unique des logiciels russes (décret du Gouvernement N° 1236).

## 4. Méthodologie des essais

### 4.1. Essais fonctionnels

#### Préparation
1. Un environnement de test est créé avec des mocks des systèmes externes (ESIA, EBS, EGRN, BdR).
2. Un jeu d'utilisateurs de test avec différents statuts KYC est préparé.
3. Des actifs et AFN de test dans divers états sont préparés.

#### Conduite
- Tests automatisés via Cypress / Playwright pour le mode PWA.
- Capacitor Test Kit (harnais Appium sur le wrapper natif) pour les builds iOS/Android.
- WebView2 Test Harness pour la build Windows.
- XCUITest pour les extensions natives iOS (biométrie, Secure Enclave, canal GOST).
- Espresso pour les extensions natives Android (Strongbox, BiometricPrompt, SZI).

#### Critères de succès
- Couverture 100 % des scénarios FT-001 — FT-020 (fonctionnels).
- Couverture 100 % des scénarios FT-021 — FT-025 (cryptographie et protection, MASVS L2+).
- 0 erreurs critiques.
- Pas plus de 5 remarques cosmétiques.

### 4.2. Essais de charge

#### Préparation
- Environnement de test similaire à la production.
- JMeter / Yandex Tank pour la charge serveur.
- Lighthouse / WebPageTest pour les métriques client.

#### Scénarios
- 5 000 RPS sur la passerelle API pendant 30 minutes.
- 50 000 utilisateurs simultanés.
- Inscription de 1 000 nouveaux utilisateurs en 5 minutes.
- Émission de 100 AFN en une heure.

#### Critères de succès
- Temps de réponse P95 ≤ 200 ms.
- Error rate ≤ 0,1 %.
- Cold start ≤ 2 secondes en 4G.
- Memory peak ≤ 300 Mo.

### 4.3. Essais de sécurité

#### Types de vérifications
- Analyse statique du code (SAST) : PT Application Inspector, PVS-Studio.
- Analyse dynamique (DAST) : PT BlackBox.
- Analyse des composants (SCA) : détection des bibliothèques vulnérables.
- Pentest externe par un laboratoire accrédité.
- Red Team interne.
- Vérification de la conformité à OWASP MASVS L2+.

#### Critères de succès
- 0 vulnérabilités des catégories Critical et High.
- Pas plus de 5 vulnérabilités Medium avec risque accepté.
- Confirmation de la conformité à GOST R 57580.1 niveau 4.

### 4.4. Essais d'accessibilité

#### Types de vérifications
- Vérification automatique via axe-core / WAVE.
- Tests avec lecteurs d'écran (VoiceOver, TalkBack, Narrator).
- Tests avec police agrandie.
- Vérification du contraste (≥ 4,5:1 pour le texte principal).

#### Critères de succès
- Conformité à WCAG 2.2 niveau AA à 100 %.
- Tous les éléments interactifs accessibles au clavier.
- Toutes les images ont des attributs alt.
