## 1. Informations générales

### 1.1. Identifiant et nom du programme
Nom complet : « CPFSR — application mobile du débiteur sur gage ».
Identifiant : РСФСР.МП.

### 1.2. Logiciels nécessaires au fonctionnement
Le programme fonctionne de manière autonome. Environnement requis :
- système d'exploitation : iOS 15+, Android 9+, Windows 10+ ;
- navigateur web pour le mode PWA avec prise en charge du Service Worker ;
- connexion réseau pour les opérations interactives (la consultation des données mises en cache est disponible hors ligne).

### 1.3. Langages de programmation
- HTML5 / CSS3 / JavaScript (ES2020+) — code principal de la PWA.
- TypeScript — côté serveur de l'API (hors application).
- Capacitor v6+ (TypeScript) — wrapper pour l'empaquetage natif (code base unique pour iOS / Android / Windows desktop).
- Ponts natifs (via Capacitor) :
  - Kotlin / Java — plugins Android (pour extensions au-delà de l'API Web) ;
  - Swift / Objective-C — plugins iOS (Secure Enclave, biométrie) ;
  - C# / Windows App SDK — uniquement pour les plugins Windows natifs dans le paquet MSIX Capacitor (Toast, FileSystem ACL, KeyVault).

## 2. Objet fonctionnel

### 2.1. Classes de tâches résolues
Le programme appartient à la classe des applications mobiles client à vocation financière et comptable, avec intégration aux systèmes d'information d'État.

### 2.2. Informations sur les limitations fonctionnelles
- L'identification et la signature des opérations exigent un compte Gosuslugi confirmé et l'accès à EBS.
- L'utilisation complète exige un appareil doté d'un stockage protégé (Secure Enclave / Trusted Execution Environment).
- La biométrie peut être indisponible sur les appareils dépourvus des capteurs correspondants.
- Installation depuis RuStore, App Store, Google Play, Microsoft Store, ou installation PWA via le navigateur.

## 3. Description de la structure logique

### 3.1. Architecture du programme
Le programme est construit selon le principe architectural **Single Page Application** en utilisant le pattern **Clean Architecture** et une séparation en couches :

1. **Presentation Layer** — écrans et composants de l'interface utilisateur (HTML/CSS).
2. **Application Layer** — logique d'interaction des écrans, routage, gestion d'état (JavaScript).
3. **Domain Layer** — entités métier (Asset, CFA, SmartContract, User, Transaction, Notification).
4. **Data Layer** — interaction avec le stockage local (LocalStorage, IndexedDB, SQLite) et les API réseau.
5. **Infrastructure Layer** — Service Worker, push-notifications, biométrie, cryptographie.

### 3.2. Structure du programme
Le programme se compose de :
- un fichier HTML unique avec CSS et JavaScript intégrés (`index.html`) ;
- un manifest PWA (`manifest.json`) ;
- un service worker pour le mode hors ligne (`sw.js`) ;
- icônes de différentes tailles (`icons/`) ;
- ressources partagées avec le projet web de la plateforme (`../../_shared/`).

### 3.3. Liens avec d'autres programmes
- **CPFSR — Backend (API)** — interaction REST via un canal sécurisé.
- **Gosuslugi (ESIA)** — OAuth 2.1 / OpenID Connect.
- **Système biométrique unifié** — REST selon le règlement BdR N° 482-P.
- **Infrastructure de paiement NSPK (Mir)** — pour l'imputation des versements.
- **SMEV 3** — via le côté serveur.

## 5. Appel et chargement

### 5.1. Installation
La description détaillée des procédures d'installation est fournie dans le Manuel d'installation et d'administration (РСФСР.МП.05).

Voies d'installation de base :
- **RuStore (Android)** — canal principal pour les utilisateurs russes.
- **App Store (iOS)** — sous réserve d'approbations réglementaires.
- **Google Play (Android)** — sous réserve d'approbations réglementaires.
- **Microsoft Store (Windows)** — sous réserve d'approbations réglementaires.
- **PWA** — installation depuis le navigateur via le dialogue « Ajouter à l'écran d'accueil ».

### 5.2. Lancement
Le lancement s'effectue via le mécanisme standard du système d'exploitation (tap sur l'icône de l'application).

### 5.3. Terminaison
La terminaison se produit lors de la mise en arrière-plan de l'application. La session utilisateur est sauvegardée dans un stockage protégé et, lors du lancement suivant, est restaurée avec exigence d'authentification biométrique ou par PIN.

## 6. Données en entrée

### 6.1. Sources
- Actions de l'utilisateur (taps, saisie de texte).
- Données issues d'ESIA lors de l'autorisation.
- Confirmations biométriques via EBS.
- Push-notifications du côté serveur.
- Données issues d'EGRN, Rospatent, systèmes dépositaires (via le côté serveur).

### 6.2. Formats
- JSON pour tous les échanges réseau.
- Formats standards de paiements et notifications selon SMEV 3.
- Confirmations biométriques selon les formats d'EBS.

## 7. Données en sortie

### 7.1. Destinataires
- Côté serveur de CPFSR (soumission d'actif, émission d'AFN, signatures, demandes de versement).
- Gosuslugi (pour la réception des notifications officielles).
- Utilisateur (visualisation à l'écran).

### 7.2. Formats
- JSON pour les requêtes API.
- Exports PDF de documents (contrats, attestations).
- Excel/CSV — export de l'historique et des revenus.

## 8. Exemple de contrôle

Le scénario complet de vérification est fourni dans le document « Programme et méthodologie d'essais » (РСФСР.МП.10).

Scénario de base abrégé :
1. Lancement de l'application.

3. Soumission d'un actif en garantie.
