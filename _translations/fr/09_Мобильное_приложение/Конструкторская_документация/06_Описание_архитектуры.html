Description de l’architecture de l’application mobile

CPFSR — application mobile

Identifiant : РСФСР.МП.06.ОА

1. Aperçu architectural

1.1. Approche

L’application est construite selon le principe PWA-first avec empaquetage dans des coquilles natives via Capacitor 6. Cela assure une code base unique pour le web, Android, iOS et Windows tout en préservant l’accès aux API natives de chaque plateforme.

1.2. Avantages de l’approche

  • Une code base unique — économie de 60–70 % des ressources de développement.
  • Possibilité de releases indépendants des plateformes.
  • PWA à part entière pour les utilisateurs sans installation.
  • Accès aux API natives via les plugins Capacitor.
  • Conformité aux exigences de substitution aux importations (sans dépendance à des frameworks cross-platform étrangers).

3. Structure des composants

3.1. Presentation Layer

  • Écrans — 23 écrans en tant que sections HTML séparées avec attributs data-screen uniques.
  • Composants — éléments UI réutilisables (cartes, boutons, formulaires, graphiques).
  • Thèmes — quatre thèmes via CSS Custom Properties.
  • i18n — prise en charge de quatre langues via dictionnaires JSON.

3.2. Application Layer

  • Router — navigation par hash d’URL (#screen-name). Prise en charge de l’historique du navigateur et du bouton « Retour » de l’OS.
  • State Manager — stockage centralisé de l’état utilisateur, des actifs, des AFN, de l’historique, des notifications.
  • Theme Manager — bascule des thèmes avec sauvegarde dans LocalStorage.
  • i18n Manager — localisation de l’interface avec changement automatique de la direction du texte pour l’arabe.
  • A11y Manager — gestion de la taille de police, du thème contrasté, du focus.

3.3. Domain Layer

Entités métier : - User — utilisateur avec identifiant, nom complet, statut KYC. - Asset — actif avec type, caractéristiques, évaluation, statut. - Valuation — évaluation avec méthode, prix, durée de validité, assurance. - SmartContract — contrat intelligent avec conditions, finalité, taux, statut. - CFA — actif financier numérique lié à un contrat intelligent, versements. - Transaction — opération dans le registre (émission, échange, versement). - Notification — notification avec catégorie, statut, action.

3.4. Data Layer

  • Repositories — abstraction d’accès aux données avec bascule entre stockage local et API réseau.
  • API Client — client REST API typé avec logique de retry et cache.
  • Local Storage — LocalStorage pour les paramètres, IndexedDB pour le cache des données, stockage protégé de l’OS pour les secrets.

3.5. Infrastructure

  • Service Worker — mise en cache des ressources statiques et des réponses API ; traitement des push-notifications ; fonctionnement en mode hors ligne.
  • Crypto — wrapper sur l’API WebCrypto + plugin natif pour la cryptographie GOST (KriptoPro Mobile SDK).
  • Network — Fetch API avec rafraîchissement automatique des tokens ; certificate pinning via plugins natifs.

6. Sécurité de l’information

6.1. Protection de l’application

  • Anti-tampering — vérification d’intégrité du code au lancement.
  • Anti-debug — détection de débogueurs et émulateurs.
  • Détection Jailbreak/Root — refus de fonctionnement sur appareils compromis.
  • Certificate pinning — épinglage des certificats CA pour la protection contre MITM.
  • Anti-screenshot — pour les écrans critiques (signature, contrat intelligent).
  • Obfuscation — obfuscation du code JS lors du build.

6.2. Protection des données

  • Chiffrement at rest — toutes les données sur l’appareil sont chiffrées avec une clé du stockage protégé.
  • Chiffrement in transit — TLS avec certificats GOST.
  • Pas de stockage de biométrie — uniquement identifiants pour la vérification via EBS.
  • Token rotation — rafraîchissement des tokens d’autorisation toutes les 15 minutes.

6.3. Conformité aux standards

  • OWASP MASVS L2+.
  • GOST R 57580.1 niveau 4.
  • FZ-152 (protection des données personnelles).
  • FZ-187 (IIC, catégorie 1).
  • Règlement de la Banque de Russie N° 683-P.

7. Performance

7.1. Budget de performance

Métrique Budget
Cold start ≤ 2 secondes en 4G
Time to Interactive (TTI) ≤ 3 secondes
Largest Contentful Paint ≤ 1,5 seconde
Cumulative Layout Shift ≤ 0,05
Bundle size (initial) ≤ 200 Ko
Transfert total ≤ 1 Mo au premier lancement
Memory footprint ≤ 300 Mo
Batterie en arrière-plan minimal (no background tasks)

7.2. Optimisations

  • Lazy loading des écrans et composants.
  • Minification et tree-shaking du JS.
  • Compression des images (WebP).
  • Mise en cache des réponses API via Service Worker.
  • Utilisation de requestIdleCallback pour les tâches non critiques.

8. Accessibilité

  • WCAG 2.2 niveau AA (obligatoire), AAA (cible).
  • Prise en charge de VoiceOver, TalkBack, Narrator.
  • Agrandissement dynamique de la police.
  • Thème contrasté pour les malvoyants.
  • Taille minimale de la zone tactile 44 × 44 pixels.
  • Attributs ARIA corrects pour tous les éléments interactifs.
  • Skip links pour navigation rapide.

9. Localisation

L’application prend en charge quatre langues : - Russe (par défaut). - Anglais. - Français. - Arabe (avec bascule automatique en mode RTL).

Toutes les chaînes sont extraites dans un dictionnaire JSON centralisé (_shared/i18n.js). L’ajout d’une nouvelle langue — ajout d’un objet au dictionnaire.

10. Déploiement

10.1. Build et empaquetage

La description détaillée du processus de build est fournie dans le README des plateformes correspondantes : - Сборки/android/README.md - Сборки/ios/README.md - Сборки/windows/README.md

10.2. CI/CD

Schéma recommandé d’intégration / livraison continues : - Dépôt Git sur une plateforme nationale (GitFlic / GitVerse). - CI à chaque commit : lint, tests unitaires, scan SAST, build. - CD sur tags de version : publication automatique vers RuStore et autres stores. - Distribution des builds bêta via RuStore Beta et TestFlight.
