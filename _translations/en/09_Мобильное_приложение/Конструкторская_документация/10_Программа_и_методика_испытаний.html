Testing Programme and Methodology

Testing Programme and Methodology

Автор: Кагиров А.-Х.А. · Свидетельство о депонировании №4011265 от 19.12.2024 · Center Group Company · Engagement: CPFSR-RF-DESIGN-2026

Testing programme and methodology

CPFSR — mobile application

Identifier: РСФСР.МП.10.ПМИ Standard: GOST 19.301-79

1. Object of testing

1.1. Name

“CPFSR — mortgagor’s mobile application” in all target builds: - Android (RuStore, Google Play); - iOS (App Store); - Windows (Microsoft Store); - PWA (any modern browser).

1.2. Purpose of testing

Confirmation that the programme complies with the requirements of the Terms of Reference (РСФСР.МП.01.ТЗ), with regulators’ normative requirements and with software product quality standards.

2. Types of testing

2.1. Functional tests

Verification of compliance with the functional requirements of the TOR. Object — all 23 screens and all transitions.

2.2. Load tests

Verification of performance under peak loads.

2.3. Security tests

Certification testing under the requirements of FSTEC of Russia, FSB of Russia.

2.4. Accessibility tests

Verification of compliance with WCAG 2.2 level AA.

2.5. Compatibility tests

Verification of operation on various device models and OS versions.

2.6. Regression tests

Automatic tests on every release.

2.3. Certification testing of cryptography and protection (MASVS L2+)

Introduced to enforce the requirements of OWASP MASVS v2.0 level L2 (for applications in the financial infrastructure) and the regulations of FSB of Russia on the use of cryptographic information security tools.

ID Scenario Description Expected result MASVS binding
FT-021 Two-level certificate pinning (MITM attack) On the MITM proxy side, the API-domain certificate is substituted. The test is performed sequentially for the primary pin and the backup pin. Requests are rejected with error SSL_PINNING_VIOLATION; the event is logged in the local security log and asynchronously to the operator’s SIEM. MASVS-NETWORK-1, L2
FT-022 Root / Jailbreak detection On Android, RootBeer is used (5 checks: su-binary, dangerous apps, busybox, build tags, RW system); on iOS — IOSSecuritySuite (jailbroken file paths, suspicious symlinks, can write outside sandbox, fork() returns success, dynamic linker hooks). On detection of root/jailbreak — blocking of access to criticality functions (DFA issue, smart contract signing); the user is shown a screen with an explanation and a support contact. MASVS-RESILIENCE-1, L2
FT-023 GOST channel payload (Magma / Kuznyechik tunnel) All calls to the API gateway are wrapped in a GOST TLS tunnel (RFC 9189) with cipher suites under GOST R 34.12-2015. The handshake, message exchange and rotation of session keys are tested. The handshake completes successfully; session keys are rotated every 60 minutes / 1 GB of traffic; the payload cannot be decrypted without a valid session key. MASVS-NETWORK-2, L2
FT-024 Secure Enclave / Strongbox for private keys The UKEP private key is stored in Secure Enclave (iOS) / Strongbox-backed KeyStore (Android API ≥ 28). Test: attempt to export the private key via debug channel, root exploit, physical access. The key does not leave the hardware module; export attempts return KEY_EXPORT_DENIED; a biometric prompt is required on each signing. MASVS-CRYPTO-1, MASVS-PLATFORM-2, L2
FT-025 Verification of the cryptographic information security tool signature On the application side, the signature of API gateway responses under GOST R 34.10-2012 is verified (key length 256/512). Test: valid signature, substituted signature, expired certificate. Valid — the response is accepted; substituted — INVALID_SIGNATURE + blocking; expired certificate — CERT_EXPIRED + chain refresh trigger. MASVS-CRYPTO-2, L2

Coverage of FT-021 … FT-025 is mandatory for passing voluntary certification in the FSB of Russia cryptographic information security tool certification system and for submission to the DFA IS operators register (779-P / Instruction 4336-U).

3.2. Stage 2. Beta testing (closed)

A closed circle of users of up to 1,000 people, over 30 days. Metrics: - Number of completed user scenarios. - Average time to complete each scenario. - Number of errors per 100 operations. - Crash-free sessions (target: ≥ 99.8%).

3.3. Stage 3. Acceptance tests

With the participation of the customer, the regulator (Bank of Russia), representatives of the relevant ministries. Verification of compliance with all TOR requirements.

3.4. Stage 4. Certification testing

Information security tools certification

  • Attestation under FSTEC of Russia Order No. 239 (CII).
  • Attestation under FSTEC of Russia Order No. 21 (personal data, UZ-1).
  • Compliance with GOST R 57580.1 level 4.
  • Certification of cryptographic means by FSB of Russia.

Software certification

  • Inclusion in the Unified Register of Russian Software (RF Government Resolution No. 1236).

4. Testing methodology

4.1. Functional tests

Preparation

  1. A test environment is created with mocks of external systems (ESIA, EBS, EGRN, CBR).
  2. A set of test users with different KYC statuses is prepared.
  3. Test assets and DFAs in various states are prepared.

Conduct

  • Automated tests via Cypress / Playwright for the PWA mode.
  • Capacitor Test Kit (Appium harness over the native wrapper) for iOS/Android builds.
  • WebView2 Test Harness for the Windows build.
  • XCUITest for iOS native extensions (biometrics, Secure Enclave, GOST channel).
  • Espresso for Android native extensions (Strongbox, BiometricPrompt, cryptographic information security tools).

Success criteria

  • 100% coverage of scenarios FT-001 — FT-020 (functional).
  • 100% coverage of scenarios FT-021 — FT-025 (cryptography and protection, MASVS L2+).
  • 0 critical errors.
  • No more than 5 cosmetic observations.

4.2. Load tests

Preparation

  • A test environment similar to production.
  • JMeter / Yandex Tank for server load.
  • Lighthouse / WebPageTest for client metrics.

Scenarios

  • 5,000 RPS at the API gateway over 30 minutes.
  • 50,000 concurrent users.
  • Registration of 1,000 new users within 5 minutes.
  • Issue of 100 DFAs within an hour.

Success criteria

  • P95 response time ≤ 200 ms.
  • Error rate ≤ 0.1%.
  • Cold start ≤ 2 seconds on 4G.
  • Memory peak ≤ 300 MB.

4.3. Security tests

Types of checks

  • Static code analysis (SAST): PT Application Inspector, PVS-Studio.
  • Dynamic analysis (DAST): PT BlackBox.
  • Software composition analysis (SCA): detection of vulnerable libraries.
  • External pentest by an accredited laboratory.
  • Internal Red Team.
  • Verification of compliance with OWASP MASVS L2+.

Success criteria

  • 0 vulnerabilities of Critical and High categories.
  • No more than 5 Medium vulnerabilities with accepted risk.
  • Confirmation of compliance with GOST R 57580.1 level 4.

4.4. Accessibility tests

Types of checks

  • Automated check via axe-core / WAVE.
  • Testing with screen readers (VoiceOver, TalkBack, Narrator).
  • Testing with enlarged font.
  • Contrast verification (≥ 4.5:1 for body text).

Success criteria

  • 100% compliance with WCAG 2.2 level AA.
  • All interactive elements are accessible from the keyboard.
  • All images have alt attributes.

4.5. Compatibility tests

Device pool

Android: - Samsung Galaxy S22, S23, S24. - Xiaomi Redmi Note 12, 13. - Realme C55, C65. - Honor 90, X9. - Pixel 7, 8.

iOS: - iPhone SE (3rd generation). - iPhone 13, 14, 15, 16. - iPad 9th generation, iPad Pro.

Windows: - Surface Pro 9, 10. - Typical workstations with Windows 10/11. - Surface Go (ARM64).

Success criteria

  • Correct operation on each device.
  • Responsive layout from 360 × 640 to 2560 × 1600.

5. Testing reporting

5.1. Content of the report

  • Functional testing protocol.
  • Load testing protocol.
  • Pentest report.
  • Certificates of compliance.
  • Register of found and remediated defects.

5.2. Storage

All testing artefacts are stored in the platform operator’s quality management system for at least 10 years (in line with the requirements of FZ-152 and the regulators).

6. Acceptance

6.1. Composition of the commission

  • Customer’s representative (A.-Kh. A. Kagirov).
  • Representative of the platform operator.
  • Representative of the Bank of Russia.
  • Representative of the Ministry of Finance / Ministry of Digital Development (invited).
  • Chief project architect.
  • Head of the IS team.

6.2. Acceptance criteria

  1. All functional requirements are fulfilled.
  2. All load requirements are fulfilled.
  3. Information security tools certification is passed.
  4. Russian software register certification is passed.
  5. RuStore publication is approved.
  6. Beta testing is completed successfully (crash-free ≥ 99.8%).

6.3. Acceptance documents

  • Acceptance test report.
  • Trial operation commissioning act.
  • Industrial operation commissioning act.

The document complies with the requirements of GOST 19.301-79 “Testing programme and methodology”.
