# Mobile application architecture description

## CPFSR — mobile application

**Identifier:** РСФСР.МП.06.ОА

---

## 1. Architectural overview

### 1.1. Approach
The application is built on the **PWA-first** principle with packaging into native shells via **Capacitor 6**. This provides a single code base for web, Android, iOS and Windows while preserving access to each platform's native APIs.

### 1.2. Advantages of the approach
- A single code base — 60–70% saving in development resources.
- The possibility of independent platform releases.
- A full-fledged PWA for users without installation.
- Access to native APIs via Capacitor plugins.
- Compliance with import substitution requirements (no dependence on foreign cross-platform frameworks).

## 3. Component structure

### 3.1. Presentation Layer
- **Screens** — 23 screens as separate HTML sections with unique data-screen attributes.
- **Components** — reusable UI elements (cards, buttons, forms, charts).
- **Themes** — four themes via CSS Custom Properties.
- **i18n** — support for four languages via JSON dictionaries.

### 3.2. Application Layer
- **Router** — navigation by URL hash (`#screen-name`). Support for browser history and the OS "Back" button.
- **State Manager** — centralised storage of user state, assets, DFAs, history, notifications.
- **Theme Manager** — theme switching with persistence in LocalStorage.
- **i18n Manager** — interface localisation with automatic text-direction switching for Arabic.
- **A11y Manager** — control of font size, high-contrast theme, focus.

### 3.3. Domain Layer
Business entities:
- `User` — a user with an identifier, full name, KYC status.
- `Asset` — an asset with type, characteristics, appraisal, status.
- `Valuation` — a valuation with method, price, validity period, insurance.
- `SmartContract` — a smart contract with conditions, target, rate, status.
- `CFA` — a digital financial asset linked to a smart contract, payments.
- `Transaction` — an operation in the register (issue, exchange, payment).
- `Notification` — a notification with category, status, action.

### 3.4. Data Layer
- **Repositories** — abstraction of data access with switching between local storage and the network API.
- **API Client** — a typed REST API client with retry logic and caching.
- **Local Storage** — LocalStorage for settings, IndexedDB for data cache, the OS protected storage for secrets.

### 3.5. Infrastructure
- **Service Worker** — caching of static resources and API responses; handling of push notifications; offline mode operation.
- **Crypto** — a wrapper over the WebCrypto API + a native plugin for GOST cryptography (KriptoPro Mobile SDK).
- **Network** — Fetch API with automatic token refresh; certificate pinning via native plugins.

## 6. Information security

### 6.1. Application protection
- **Anti-tampering** — code integrity check on launch.
- **Anti-debug** — detection of debuggers and emulators.
- **Jailbreak/Root detection** — refusal to operate on compromised devices.
- **Certificate pinning** — pinning of CA certificates to protect against MITM.
- **Anti-screenshot** — for critical screens (signing, smart contract).
- **Obfuscation** — JS code obfuscation at build time.

### 6.2. Data protection
- **At-rest encryption** — all data on the device is encrypted with a key from the protected storage.
- **In-transit encryption** — TLS with GOST certificates.
- **No biometrics storage** — only identifiers for verification via EBS.
- **Token rotation** — refresh of authorisation tokens every 15 minutes.

### 6.3. Standards compliance
- OWASP MASVS L2+.
- GOST R 57580.1 level 4.
- FZ-152 (personal data protection).
- FZ-187 (CII, category 1).
- Bank of Russia Regulation No. 683-P.

## 7. Performance

### 7.1. Performance budget

| Metric | Budget |
|---|---|
| Cold start | ≤ 2 seconds on 4G |
| Time to Interactive (TTI) | ≤ 3 seconds |
| Largest Contentful Paint | ≤ 1.5 seconds |
| Cumulative Layout Shift | ≤ 0.05 |
| Bundle size (initial) | ≤ 200 KB |
| Total transfer | ≤ 1 MB on first launch |
| Memory footprint | ≤ 300 MB |
| Battery in background | minimal (no background tasks) |

### 7.2. Optimisations
- Lazy loading of screens and components.
- Minification and tree-shaking of JS.
- Compression of images (WebP).
- Caching of API responses via Service Worker.
- Use of `requestIdleCallback` for non-critical tasks.

## 8. Accessibility

- WCAG 2.2 level AA (mandatory), AAA (target).
- Support for VoiceOver, TalkBack, Narrator.
- Dynamic font enlargement.
- High-contrast theme for the visually impaired.
- Minimum touch area size 44 × 44 pixels.
- Correct ARIA attributes for all interactive elements.
- Skip links for fast navigation.

## 9. Localisation

The application supports four languages:
- Russian (default).
- English.
- French.
- Arabic (with automatic switching to RTL layout).

All strings are extracted into a centralised JSON dictionary (`_shared/i18n.js`). Adding a new language — adding one object to the dictionary.
