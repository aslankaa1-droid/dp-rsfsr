# Programme description

## CPFSR — mobile application

**Identifier:** РСФСР.МП.02.ОП
**Standard:** GOST 19.402-78
**Rightsholder:** Abdul-Khakim Akhmadovich Kagirov

---

## 1. General information

### 1.1. Identifier and name of the programme
Full name: "CPFSR — mortgagor's mobile application".
Identifier: РСФСР.МП.

### 1.2. Software required for operation
The programme operates autonomously. Required environment:
- operating system: iOS 15+, Android 9+, Windows 10+;
- web browser for the PWA mode with Service Worker support;
- network connection for interactive operations (viewing cached data is available offline).

### 1.3. Programming languages
- HTML5 / CSS3 / JavaScript (ES2020+) — the main PWA code.
- TypeScript — the server-side of the API (outside the application).
- Capacitor v6+ (TypeScript) — wrapper for native packaging (a single code base for iOS / Android / Windows desktop).
- Native bridges (via Capacitor):
  - Kotlin / Java — Android plugins (for extensions beyond the Web API);
  - Swift / Objective-C — iOS plugins (Secure Enclave, biometrics);
  - C# / Windows App SDK — only for native Windows plugins within the Capacitor MSIX package (Toast, FileSystem ACL, KeyVault).

## 2. Functional purpose

### 2.1. Classes of tasks solved
The programme belongs to the class of client mobile applications of financial accounting purpose with integration with state information systems.

### 2.2. Information on functional limitations
- Identification and signing of operations require a confirmed Gosuslugi account and access to EBS.
- Full use requires a device with a protected storage (Secure Enclave / Trusted Execution Environment).
- Biometrics may be unavailable on devices without the relevant sensors.
- Installation from RuStore, App Store, Google Play, Microsoft Store, or a PWA installation via the browser.

## 3. Description of the logical structure

### 3.1. Programme architecture
The programme is built according to the **Single Page Application** architectural principle using the **Clean Architecture** pattern and a division into layers:

1. **Presentation Layer** — user interface screens and components (HTML/CSS).
2. **Application Layer** — screen interaction logic, routing, state management (JavaScript).
3. **Domain Layer** — business entities (Asset, CFA, SmartContract, User, Transaction, Notification).
4. **Data Layer** — interaction with local storage (LocalStorage, IndexedDB, SQLite) and network APIs.
5. **Infrastructure Layer** — Service Worker, push notifications, biometrics, cryptography.

### 3.2. Programme structure
The programme consists of:
- a single HTML file with embedded CSS and JavaScript code (`index.html`);
- a PWA manifest (`manifest.json`);
- a service worker for offline mode (`sw.js`);
- icons of various sizes (`icons/`);
- shared resources with the platform's web project (`../../_shared/`).

### 3.3. Links to other programmes
- **CPFSR — Backend (API)** — REST interaction via a secured channel.
- **Gosuslugi (ESIA)** — OAuth 2.1 / OpenID Connect.
- **Unified Biometric System** — REST under CBR Regulation No. 482-P.
- **NSPK (Mir) payment infrastructure** — for crediting payments.
- **SMEV 3** — via the server-side.

## 5. Launch and loading

### 5.1. Installation
Detailed installation procedures are provided in the Installation and Administration Manual (РСФСР.МП.05).

Basic installation paths:
- **RuStore (Android)** — the primary channel for Russian users.
- **App Store (iOS)** — subject to regulatory approvals.
- **Google Play (Android)** — subject to regulatory approvals.
- **Microsoft Store (Windows)** — subject to regulatory approvals.
- **PWA** — installation from the browser via the "Add to home screen" dialog.

### 5.2. Launch
Launching is performed through the standard operating system mechanism (tapping the application icon).

### 5.3. Termination
Termination occurs when the application is minimised. The user's session is saved in a protected storage and is restored on the next launch with the requirement of biometric or PIN authentication.

## 6. Input data

### 6.1. Sources
- User actions (taps, text input).
- Data from ESIA upon authorisation.
- Biometric confirmations via EBS.
- Push notifications from the server-side.
- Data from EGRN, Rospatent, depositary systems (via the server-side).

### 6.2. Formats
- JSON for all network exchanges.
- Standard payment and notification formats per SMEV 3.
- Biometric confirmations per EBS formats.

## 7. Output data

### 7.1. Addressees
- Server-side of CPFSR (asset submission, DFA issue, signing, payout requests).
- Gosuslugi (for the receipt of official notifications).
- User (visualisation on the screen).

### 7.2. Formats
- JSON for API requests.
- PDF exports of documents (contracts, statements).
- Excel/CSV — export of history and income.

## 8. Test case

The full verification scenario is provided in the document "Testing programme and methodology" (РСФСР.МП.10).

A shortened basic scenario:
1. Launching the application.
2. Authorisation via ESIA (simulated in demonstration mode).
3. Submission of an asset as collateral.
4. Receipt of the valuation result.
5. Configuration and signing of the smart contract.
6. Verification of the DFA issue.
7. Verification of the display of the issued DFA in the portfolio.
8. Verification of the receipt of the issue notification.
