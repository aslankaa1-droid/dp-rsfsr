## 1. Purpose of the document

The document sets out the requirements for the operating environment of the "CPFSR" mobile application (hereinafter — the Application) across all target platforms: Android, iOS, Windows, web PWA. The document is mandatory for:
- FSTEC / FSB certification;
- categorisation of the CII object (RF Government Resolution No. 127);
- submission to the Bank of Russia DFA IS operators register (Regulation No. 779-P);
- deployment on distribution platforms (RuStore, App Store, Microsoft Store).

## 4. Communication channel requirements

### 4.1. Bandwidth

| Scenario | Minimum | Recommendation |
|---|---:|---:|
| Authorisation via ESIA (demo/prod) | 1 Mbit/s | 5 Mbit/s |
| Loading documents from EGRN (via SMEV 3) | 2 Mbit/s | 10 Mbit/s |
| Biometric confirmation via EBS | 5 Mbit/s | 25 Mbit/s |
| Portfolio viewing scenario | 1 Mbit/s | 5 Mbit/s |
| Smart contract signing with UKEP | 2 Mbit/s | 10 Mbit/s |
| Offline viewing (local cache only) | — | — |

### 4.2. Connection quality requirements

- Acceptable latency (RTT) to the server node: no more than 800 ms.
- Acceptable packet loss: up to 1%.
- Acceptable offline periods: the Application's offline mode preserves the latest data in a local protected storage (Secure Storage / Keychain); synchronisation occurs upon connection restoration via background sync.

### 4.3. Communication channel security

- HTTPS only (TLS 1.2+); cleartext prohibited (`allowMixedContent: false`).
- Certificate pinning with two pins (primary + backup) for the `dp-rsfsr.ru` domain (see `Сборки/android/README.md`, `Сборки/ios/README.md`).
- Support for GOST TLS (CryptoPro CSP) for integration with the Bank of Russia's cryptographic information security tool channels.

## 5. Operator data centre infrastructure requirements

### 5.1. Target infrastructure (required for production)

| Parameter | Value |
|---|---|
| Number of data centres (geographically distributed placement) | ≥ 3 (including one holding center) |
| Data centre tier (Uptime Institute / GOST R 51275) | Tier-III or equivalent |
| Power redundancy | N+1 (for UPS and generators) |
| Cooling redundancy | N+1 |
| Network channel redundancy | 2 independent providers |
| Minimum channel bandwidth | 10 Gbit/s with scaling capability |
| Target uptime | 99.95% (≈ 4 h downtime per year) |
| RTO (Recovery Time Objective) | ≤ 4 hours |
| RPO (Recovery Point Objective) | ≤ 15 minutes |

### 5.2. Placement categorisation

- **Russian jurisdiction** — all data centres are physically located on the territory of the RF (compliance with Art. 12 of FZ-152 on the localisation of personal data of RF citizens).
- **FSTEC certification** — all sites are certified under the requirements for significant CII objects (K1) under FSTEC Order No. 239.
- **Security level UZ-1** under RF Government Resolution No. 1119.
- **Specific data centre operators** — selection by tender; target candidates at the time of preparation: "Rostelecom-DCs", DataPro, IKS-Holding, IXcellerate. The final choice is fixed at the application to the DFA IS operators register (779-P).

### 5.3. DLT infrastructure

- DLT vendor: **Masterchain** (target; alternative in case of unavailability — Waves Enterprise) — the specific choice is fixed in the application to the Bank of Russia.
- Register type: permissioned (with the CBR regulator node).
- Finality: deterministic.
- Cryptography: GOST R 34.10-2012 (signatures), GOST R 34.11-2012 (hash), GOST R 34.12-2015 (cipher).
- CBR regulator node: provided for in the architecture (see РСФСР.МП.06 "Architecture description", item 3.7).

## 8. Operating mode and maintenance requirements

### 8.1. Application operating mode

- **End user:** 24×7 (with scheduled technology windows).
- **Operator backend:** 24×7 with a target uptime of 99.95%.
- **Scheduled technology window:** Sunday night 02:00–06:00 MSK (once a month); announcement no later than 7 days before the window.
- **Emergency technology window:** allowed up to 2 hours with prior notification.

### 8.2. Update procedure

- Mobile application: automatic update via RuStore / App Store / Microsoft Store; critical security patches — within 72 hours of detection.
- Server-side: blue-green deployment, without downtime.

### 8.3. Backup

- Full nightly database backup.
- Backup storage — in a separate geolocation (≥ 500 km from the main data centre).
- Cryptographic encryption of backups with GOST algorithms.
- Regular restore testing (DR drill) — at least once per quarter.

## 9. Special requirements

### 9.1. Operating environment security

- The end user's device must support a hardware TEE (Trusted Execution Environment) for the storage of the user's cryptographic keys. Minimum requirements:
  - Android: Strongbox / TEE Keymaster (API 28+).
  - iOS: Secure Enclave.
  - Windows: TPM 2.0.
- On devices with jailbreak / root, the application operates in a limited mode (history viewing only) with a warning.

### 9.2. Sensitivity to environment quality

- If the TLS level drops below 1.2 — connection is refused.
- On detection of an MITM attack (mismatch with the pinned certificate) — block with CISO notification.
- On channel degradation below 1 Mbit/s — automatic switch to lite mode (text data only).
