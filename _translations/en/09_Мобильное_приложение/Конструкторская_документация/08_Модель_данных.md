# Mobile application data model

## CPFSR

**Identifier:** РСФСР.МП.08.МД

## 1. Local data model (on the device)

### 1.1. Storage

| Data type | Storage | Encryption |
|---|---|---|
| Tokens, keys, biometric identifiers | iOS Keychain / Android Keystore / Windows Credential Locker | by OS means |
| Application settings | LocalStorage | none (non-sensitive) |
| Data cache | IndexedDB / SQLite | AES-256 with a local key |
| Temporary files | App Sandbox | by OS means |

## 2. Server-side data model

The full data model is described in the platform's technical documentation (`01_Техническая_документация/06_DLT_реестр_ЦФА.md`). The client uses REST projections of the server-side model.

### Key entities
- **Identity** (user) — link to ESIA.
- **KycRecord** — KYC check records.
- **Asset** — an asset linked to EGRN/Rospatent/depositary.
- **Valuation** — a valuation with partner, method, insurance.
- **SmartContract** — a smart contract with the signatures of all parties.
- **CFAToken** — a token in the DLT register with full history.
- **Escrow** — a Bank of Russia escrow record.
- **Treasury** — Federal Treasury records.
- **NationalProgramme** — a national project programme.
- **InsurancePolicy** — an insurance policy.
- **EgrnRecord** — an encumbrance record.
- **Notification** — a user notification.
- **AuditLog** — an immutable journal of all operations.

## 3. Relationships between entities

```
User ─< Asset ─< Valuation
        │
        └─< SmartContract ──> CFAToken ──> Escrow ──> Treasury
                                  │
                                  ├─> InsurancePolicy
                                  ├─> EgrnRecord
                                  └─< Transaction (payments)

User ─< Notification
User ─< AuditLog (read-only)
```

## 4. Data life cycle

### 4.1. Asset

```
DRAFT → PENDING_VALUATION → VALUED → CONTRACT_PENDING →
  ACTIVE → [RETURNED | REINVESTED | EXCHANGED]
```

### 4.2. CFA

```
PENDING_ISSUE → ISSUED → IN_CB_PORTFOLIO → ACTIVE →
  [MATURED → RETURNED | EXCHANGED_MENA | INSURED_CASE | REINVESTED]
```

## 6. Backup

User data are automatically replicated in the server-side. The device stores only a cache for the offline mode. Upon a change of device, all data are restored after the first authorisation on the new device.

## 7. Deletion of user data

On deletion of the user account in line with FZ-152:
1. Data on the device — deleted immediately.
2. Data in the operational DB — deleted within 30 days.
3. Data in the DLT register (for active DFAs) — the encumbrance must be removed; thereafter, the register record remains in archived form marked with the deletion of the owner's personal data, but with the chronology preserved (supervisory reporting requirement).
4. Backups — deleted on the rotation schedule (up to 7 years of storage as required by the CBR).
