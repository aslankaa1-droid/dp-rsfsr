Non-Functional Requirements

Автор: Кагиров А.-Х.А. · Свидетельство о депонировании №4011265 от 19.12.2024 · Center Group Company · Engagement: CPFSR-RF-DESIGN-2026

Non-functional requirements (NFR) of CPFSR

The document elaborates the requirements for performance, reliability, information security, observability and operation of the platform.

1. Performance

Indicator Target value Measurement method
Peak input load 5,000 RPS at the API gateway Load testing (JMeter / Yandex Tank)
Average business API response time ≤ 200 ms (P95) Prometheus / OpenTelemetry
Response time of “heavy” operations (DFA issue) ≤ 30 sec E2E Synthetic monitoring
Time to the first screen of the mobile application ≤ 1.5 sec on 4G Real User Monitoring
TPS of the DLT register ≥ 200 transactions/sec Profiling of the DLT node
Concurrent users ≥ 50,000 simultaneously Stress testing

2. Reliability and availability

Indicator Target value
Annual SLA 99.99% (≤ 52 min downtime/year)
MTTR (mean time to repair) ≤ 15 min for critical circuits
MTBF (mean time between failures) ≥ 8,760 h for critical components
RPO (recovery point) 0 for the DLT register and DFA data; ≤ 5 min for the rest
RTO (recovery time) ≤ 5 min for critical, ≤ 30 min for the rest
Geo-distribution 3 active-active data centres
DB redundancy Synchronous replication ≥ N+2
Backup Hourly incremental + daily full, retention 7 years

3. Scalability

  1. Horizontal scaling of all stateless services via Kubernetes HPA.
  2. Vertical scaling of stateful components (DB) — up to 1 PB of storage per cluster.
  3. Sharding of the DFA register by the mortgagor’s identifier for large volumes.
  4. Partner distribution model for the load of the valuation and insurance modules.

4. Information security

4.1. Protection level

The platform complies with the 4th (maximum) level of protection under GOST R 57580.1 and implements all 8 areas of protection: - OZD — data protection; - OPI — protection of the IS perimeter; - RSI — access control and segregation; - MAU — monitoring, audit, response; - OPT — provision of continuous protection; - UPI — incident management; - OZU — protection of accounts; - RTsK — response to computer attacks.

4.2. Cryptography

  • Only certified means: KriptoPro CSP 5.0, ViPNet CSP, KriptoPro HSM.
  • Algorithms: GOST R 34.10-2012 (signature), GOST R 34.11-2012 (hash), GOST R 34.12-2015 / 34.13-2015 (encryption).
  • TLS channels — using GOST certificates (KriptoPro TLS).

4.3. Personal data protection

  • Compliance with FZ-152 and FSTEC Order No. 21.
  • ISPDn protection level — UZ-1 (under RF Government Resolution No. 1119).
  • Registration in the register of personal data operators of Roskomnadzor.
  • All operations with personal data are journalised in the immutable audit log.

4.4. CII protection

  • Significance category — 1st (highest) (FZ-187).
  • Connection to GosSOPKA.
  • Compliance with FSTEC Order No. 239.
  • Continuous interaction with NCCCI.

5. Observability

5.1. Metrics

  • Business metrics: number of issued DFAs, issue volume, issue time, conversion.
  • Technical metrics: RPS, latency, error rate, resource saturation.
  • IS metrics: number of incidents, blocked requests, failed authorisations.
  • Metrics retention — 13 months in VictoriaMetrics with aggregation by periods.

5.2. Logs

  • Structured JSON logs of all services.
  • Centralised collection in Loki / a domestic analogue.
  • Correlation via trace_id.
  • Retention of operational logs — 13 months.
  • Retention of audit logs — 10 years (in line with the requirements of FZ-152 and the Tax Code of the RF).

5.3. Tracing

  • OpenTelemetry stack across all services.
  • 10% sampling for regular requests, 100% for critical circuits (DFA issue, exchange).
  • Trace retention — 30 days (HOT), 180 days (COLD).

5.4. Alerting

  • Two-tier system: automated alerts + 24/7 SOC duty shift.
  • Escalation: L1 (1 min) → L2 (5 min) → L3 (15 min) → operator management (30 min).

6. Operation

6.1. CI/CD

  • Git repository — domestic GitFlic / GitVerse / proprietary GitLab on-prem.
  • CI: pipelines with mandatory steps:
    1. Static code analysis (PT Application Inspector, PVS-Studio).
    2. SAST / DAST (PT AppSec).
    3. Composition analysis (detection of vulnerable libraries).
    4. Unit tests (coverage ≥ 80%).
    5. Integration tests.
    6. Performance smoke tests.
    7. UKEP-signing of the release build.
  • CD: GitOps (Argo CD / a domestic analogue), full automation of deployments, with a canary release mechanism.

6.2. Configuration management

  • All configurations — in Git (Infrastructure as Code, Configuration as Code).
  • Secrets — through HashiCorp Vault (or a domestic analogue) with encryption on HSM.

6.3. Backup and DR

  • Full DR scenario testing — quarterly.
  • Recovery drill — once every six months with the participation of the SOC.

7. Compatibility and standardisation

  • API: REST (OpenAPI 3.1) for external integrations; gRPC between internal services.
  • API documentation — automatic generation in the Swagger style.
  • Standards support: OAuth 2.1, OpenID Connect, mTLS, JWT with RS256/GostR3410.
  • Compliance with ISO/IEC 25010 (software quality attributes).

8. Requirements for the teams and operation

  • 24×7 NOC (Network Operations Center).
  • 24×7 SOC (Security Operations Center) with GosSOPKA integration.
  • Duty shifts: ≥ 6 people per shift for critical circuits.
  • Regular Red Team / Blue Team practices — every 6 months.
  • Annual external pentest by a laboratory accredited by FSTEC.

9. Requirements for import independence

100% of the components of the hardware and software stack must: - be included in the Unified Register of Russian Software (RF Government Resolution No. 1236), or - be certified by FSTEC of Russia / FSB of Russia, or - be a proprietary development of the operator with open source and a build on the territory of the RF.
