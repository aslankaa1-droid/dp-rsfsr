# وصف البرنامج

## CPFSR — تطبيق الهاتف المحمول

**الرمز:** РСФСР.МП.02.ОП
**المعيار:** GOST 19.402-78
**صاحب الحقوق:** عبد الحكيم أحمدوفيتش كاجيروف

---

## 1. معلومات عامة

### 1.1. رمز البرنامج واسمه
الاسم الكامل: «CPFSR — تطبيق الهاتف المحمول للراهن».
الرمز: РСФСР.МП.

### 1.2. البرمجيات اللازمة للتشغيل
يعمل البرنامج باستقلالية. البيئة المطلوبة:
- نظام التشغيل: iOS 15+، Android 9+، Windows 10+؛
- متصفح ويب لوضع PWA يدعم Service Worker؛
- اتصال شبكي للعمليات التفاعلية (يتاح عرض البيانات المخزَّنة مؤقتاً دون اتصال).

### 1.3. لغات البرمجة
- HTML5 / CSS3 / JavaScript (ES2020+) — الشيفرة الأساسية لـ PWA.
- TypeScript — الجزء الخادم من API (خارج التطبيق).
- Capacitor v6+ (TypeScript) — غلاف للتحزيم الأصلي (قاعدة شيفرة موحَّدة لـ iOS / Android / Windows desktop).
- جسور أصلية (عبر Capacitor):
  - Kotlin / Java — إضافات Android (للتوسعات خارج Web API)؛
  - Swift / Objective-C — إضافات iOS (Secure Enclave، البيومترية)؛
  - C# / Windows App SDK — فقط للإضافات الأصلية لنظام Windows ضمن حزمة MSIX لـ Capacitor (Toast، FileSystem ACL، KeyVault).

## 2. الغرض الوظيفي

### 2.1. فئات المهام المحلولة
ينتمي البرنامج إلى فئة تطبيقات الهاتف المحمول العميلة ذات الغرض المالي والمحاسبي بتكامل مع نظم المعلومات الحكومية.

### 2.2. معلومات عن القيود الوظيفية
- يستلزم التعريف وتوقيع العمليات حساب Gosuslugi مؤكَّداً والوصول إلى EBS.
- يستلزم الاستخدام الكامل جهازاً بمخزن محمي (Secure Enclave / Trusted Execution Environment).
- قد لا تتوفر البيومترية على الأجهزة التي تفتقر إلى المستشعرات المناسبة.
- التثبيت من RuStore وApp Store وGoogle Play وMicrosoft Store، أو تثبيت PWA عبر المتصفح.

## 3. وصف البنية المنطقية

### 3.1. بنية البرنامج
يُبنى البرنامج وفق المبدأ المعماري **Single Page Application** باستخدام نمط **Clean Architecture** والتقسيم إلى طبقات:

1. **Presentation Layer** — شاشات ومكوّنات واجهة المستخدم (HTML/CSS).
2. **Application Layer** — منطق تفاعل الشاشات والتوجيه وإدارة الحالة (JavaScript).
3. **Domain Layer** — كيانات الأعمال (Asset، CFA، SmartContract، User، Transaction، Notification).
4. **Data Layer** — التفاعل مع التخزين المحلي (LocalStorage، IndexedDB، SQLite) وواجهات API الشبكية.
5. **Infrastructure Layer** — Service Worker، إشعارات push، البيومترية، التشفير.

### 3.2. هيكل البرنامج
يتكوّن البرنامج من:
- ملف HTML واحد بشيفرة CSS وJavaScript مدمجة (`index.html`)؛
- بيان PWA (`manifest.json`)؛
- service worker لوضع عدم الاتصال (`sw.js`)؛
- أيقونات بأحجام مختلفة (`icons/`)؛
- موارد مشتركة مع المشروع الإلكتروني للمنصة (`../../_shared/`).

### 3.3. الروابط مع برامج أخرى
- **CPFSR — Backend (API)** — تفاعل REST عبر قناة مؤمَّنة.
- **Gosuslugi (ESIA)** — OAuth 2.1 / OpenID Connect.
- **النظام البيومتري الموحَّد** — REST وفق نظام بنك روسيا رقم 482-P.
- **بنية مدفوعات NSPK (Mir)** — لقيد المدفوعات.
- **SMEV 3** — عبر الجزء الخادم.

## 4. الوسائل التقنية المستخدمة

### الجزء الخادم (خارج التطبيق)
موصوف في وثيقة «مفهوم البنية المعمارية» لمشروع CPFSR (`01_Техническая_документация/01_Концепция_архитектуры.md`).

### أجهزة العميل
- iPhone، iPad بدعم iOS 15 وأعلى.
- أجهزة Android من الفئة المتوسطة وأعلى (من 3 جيجابايت من الذاكرة التشغيلية).
- أجهزة سطح المكتب واللوحية تحت Windows 10/11.
- أي جهاز بمتصفح ويب حديث لوضع PWA.

## 5. الاستدعاء والتحميل

### 5.1. التثبيت
الوصف التفصيلي لإجراءات التثبيت متوفر في دليل التثبيت والإدارة (РСФСР.МП.05).

مسارات التثبيت الأساسية:
- **RuStore (Android)** — القناة الرئيسية للمستخدمين الروس.
- **App Store (iOS)** — عند الحصول على الموافقات التنظيمية.
- **Google Play (Android)** — عند الحصول على الموافقات التنظيمية.
- **Microsoft Store (Windows)** — عند الحصول على الموافقات التنظيمية.
- **PWA** — التثبيت من المتصفح عبر حوار «إضافة إلى الشاشة الرئيسية».

### 5.2. التشغيل
يتم التشغيل عبر الآلية المعتادة لنظام التشغيل (النقر على أيقونة التطبيق).

### 5.3. الإنهاء
يحدث الإنهاء عند تصغير التطبيق. تُحفَظ جلسة المستخدم في مخزن محمي وتُستعاد عند التشغيل التالي مع اشتراط المصادقة البيومترية أو بـ PIN.

## 6. بيانات الإدخال

### 6.1. المصادر
- إجراءات المستخدم (نقرات، إدخال نصي).
- البيانات من ESIA عند الترخيص.
- التأكيدات البيومترية عبر EBS.
- إشعارات push من الجزء الخادم.
- البيانات من EGRN وRospatent وأنظمة الإيداع (عبر الجزء الخادم).

### 6.2. الصيغ
- JSON لجميع التبادلات الشبكية.
- الصيغ المعيارية للمدفوعات والإشعارات وفق SMEV 3.
- التأكيدات البيومترية وفق صيغ EBS.

## 7. بيانات الإخراج

### 7.1. المستلمون
- الجزء الخادم لـ CPFSR (تقديم الأصل، إصدار DFA، التواقيع، طلبات الدفع).
- Gosuslugi (لاستقبال الإشعارات الرسمية).
- المستخدم (التصوير على الشاشة).

### 7.2. الصيغ
- JSON لطلبات API.
- صادرات PDF للمستندات (العقود، الشهادات).
- Excel/CSV — تصدير السجل والدخل.

## 8. المثال المرجعي

السيناريو الكامل للتحقق متوفر في وثيقة «برنامج ومنهجية الاختبارات» (РСФСР.МП.10).

السيناريو الأساسي المختصر:
1. تشغيل التطبيق.
2. الترخيص عبر ESIA (محاكاة في الوضع التوضيحي).
3. تقديم الأصل كضمان.
4. استلام نتيجة التقييم.
5. ضبط العقد الذكي وتوقيعه.
6. التحقق من إصدار DFA.
7. التحقق من ظهور DFA المُصدَر في المحفظة.
8. التحقق من تلقي إشعار الإصدار.
