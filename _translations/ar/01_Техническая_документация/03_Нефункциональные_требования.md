# المتطلبات غير الوظيفية (NFR) لمنصة CPFSR

يفصِّل المستند متطلبات الأداء والموثوقية وأمن المعلومات والقابلية للرصد وتشغيل المنصة.

---

## 1. الأداء

| المؤشر | القيمة المستهدفة | منهج القياس |
|---|---|---|
| الحِمل الأقصى للإدخال | 5 000 طلب/ثانية في بوابة API | اختبارات الحِمل (JMeter / Yandex Tank) |
| متوسط زمن استجابة API الأعمال | ≤ 200 ميلي ثانية (P95) | Prometheus / OpenTelemetry |
| زمن استجابة العمليات «الثقيلة» (إصدار DFA) | ≤ 30 ثانية E2E | الرصد التركيبي |
| زمن الشاشة الأولى لتطبيق الهاتف المحمول | ≤ 1,5 ثانية على 4G | Real User Monitoring |
| TPS لسجل DLT | ≥ 200 عملية/ثانية | تنميط عقدة DLT |
| المستخدمون المتزامنون | ≥ 50 000 في وقت واحد | اختبارات الإجهاد |

## 2. الموثوقية والتوافر

| المؤشر | القيمة المستهدفة |
|---|---|
| SLA السنوي | 99,99% (≤ 52 دقيقة من التوقف/سنة) |
| MTTR (متوسط زمن الإصلاح) | ≤ 15 دقيقة للدوائر الحرجة |
| MTBF (متوسط الزمن بين الأعطال) | ≥ 8 760 ساعة للمكوّنات الحرجة |
| RPO (نقطة الاستعادة) | 0 لسجل DLT وبيانات DFA؛ ≤ 5 دقائق للباقي |
| RTO (زمن الاستعادة) | ≤ 5 دقائق للحرجة، ≤ 30 دقيقة للباقي |
| التوزّع الجغرافي | 3 مراكز بيانات active-active |
| تكرار قاعدة البيانات | تكرار متزامن ≥ N+2 |
| النسخ الاحتياطي | تزايدي كل ساعة + كامل يومياً، الاحتفاظ 7 سنوات |

## 3. قابلية التوسّع

1. **التوسّع الأفقي** لجميع الخدمات بلا حالة عبر Kubernetes HPA.
2. **التوسّع العمودي** للمكوّنات ذات الحالة (قواعد البيانات) — حتى 1 PB من التخزين لكل عنقود.
3. **التجزئة (Sharding)** لسجل DFA حسب معرّف الراهن للأحجام الكبيرة.
4. **نموذج التوزيع الشريك** لحِمل وحدات التقييم والتأمين.

## 4. أمن المعلومات

### 4.1. مستوى الحماية
تطابق المنصة **المستوى الرابع (الأعلى) للحماية وفق GOST R 57580.1**، وتُنفِّذ جميع المجالات الثمانية للحماية:
- OZD — حماية البيانات؛
- OPI — حماية محيط نظام المعلومات؛
- RSI — الفصل والتحكم في الوصول؛
- MAU — الرصد، التدقيق، الاستجابة؛
- OPT — توفير الحماية المستمرة؛
- UPI — إدارة الحوادث؛
- OZU — حماية بيانات الاعتماد؛
- RTsK — الاستجابة للهجمات الحاسوبية.

### 4.2. التشفير
- وسائل معتمدة حصراً: KriptoPro CSP 5.0، ViPNet CSP، KriptoPro HSM.
- الخوارزميات: GOST R 34.10-2012 (التوقيع)، GOST R 34.11-2012 (التجزئة)، GOST R 34.12-2015 / 34.13-2015 (التشفير).
- قنوات TLS — باستخدام شهادات GOST (KriptoPro TLS).

### 4.3. حماية البيانات الشخصية
- المطابقة لـ FZ-152 ولأمر FSTEC رقم 21.
- مستوى أمان ISPDn — **UZ-1** (وفق قرار الحكومة رقم 1119).
- التسجيل في سجل مشغّلي البيانات الشخصية لدى Roskomnadzor.
- جميع العمليات على البيانات الشخصية تُسجَّل في سجل التدقيق غير القابل للتعديل.

### 4.4. حماية CII
- فئة الأهمية — **الأولى (الأعلى)** (FZ-187).
- الاتصال بـ GosSOPKA.
- المطابقة لأمر FSTEC رقم 239.
- التفاعل المستمر مع NCCCI.

## 5. القابلية للرصد

### 5.1. المقاييس
- مقاييس الأعمال: عدد DFA المُصدَرة، حجم الإصدار، زمن الإصدار، التحويل.
- المقاييس الفنية: RPS، الكُمون، معدل الأخطاء، إشباع الموارد.
- مقاييس أمن المعلومات: عدد الحوادث، الطلبات المحظورة، حالات الترخيص الفاشلة.
- الاحتفاظ بالمقاييس — 13 شهراً في VictoriaMetrics مع تجميع حسب الفترات.

### 5.2. السجلات
- سجلات JSON منظَّمة لجميع الخدمات.
- جمع مركزي في Loki / نظير محلي.
- الترابط عبر trace_id.
- الاحتفاظ بالسجلات التشغيلية — 13 شهراً.
- الاحتفاظ بسجلات التدقيق — 10 سنوات (يتوافق مع متطلبات FZ-152 وقانون الضرائب للاتحاد الروسي).

### 5.3. التتبّع
- حزمة OpenTelemetry لجميع الخدمات.
- أخذ عينات 10% للطلبات الاعتيادية، 100% للدوائر الحرجة (إصدار DFA، التبادل).
- الاحتفاظ بالتتبّعات — 30 يوماً (HOT)، 180 يوماً (COLD).

### 5.4. التنبيهات
- نظام بمستويَين: تنبيهات تلقائية + مناوبة SOC.
- التصعيد: L1 (دقيقة 1) → L2 (5 دقائق) → L3 (15 دقيقة) → إدارة جهة التشغيل (30 دقيقة).

## 6. التشغيل

### 6.1. CI/CD
- مستودع Git — GitFlic / GitVerse / GitLab on-prem خاص.
- CI: مسارات بخطوات إلزامية:
  1. التحليل الساكن للشيفرة (PT Application Inspector، PVS-Studio).
  2. SAST / DAST (PT AppSec).
  3. تحليل التركيب (كشف المكتبات القابلة للاستغلال).
  4. اختبارات الوحدة (التغطية ≥ 80%).
  5. اختبارات التكامل.
  6. اختبارات الأداء الأوّلية (smoke).
  7. توقيع UKEP على بناء الإصدار.
- CD: GitOps (Argo CD / نظير محلي)، أتمتة كاملة للنشر، مع آلية النشر الكناري.

### 6.2. إدارة الإعدادات
- جميع الإعدادات — في Git (Infrastructure as Code، Configuration as Code).
- الأسرار — عبر HashiCorp Vault (أو نظير محلي) مع التشفير على HSM.

### 6.3. النسخ الاحتياطي وDR
- اختبار كامل لسيناريوهات DR — كل ربع سنة.
- تمرين الاستعادة (Recovery drill) — مرة كل نصف سنة بمشاركة SOC.

## 7. التوافق والمعايير

- API: REST (OpenAPI 3.1) للتكاملات الخارجية؛ gRPC بين الخدمات الداخلية.
- توثيق API — توليد تلقائي بأسلوب Swagger.
- دعم المعايير: OAuth 2.1، OpenID Connect، mTLS، JWT مع RS256/GostR3410.
- المطابقة لـ ISO/IEC 25010 (سمات جودة البرمجيات).

## 8. متطلبات الفِرق والتشغيل

- NOC على مدار الساعة (Network Operations Center).
- SOC على مدار الساعة (Security Operations Center) مع التكامل مع GosSOPKA.
- مناوبات: ≥ 6 أشخاص لكل مناوبة للدوائر الحرجة.
- ممارسات Red Team / Blue Team بانتظام — كل 6 أشهر.
- اختبار اختراق خارجي سنوي بواسطة مختبر معتمد من FSTEC.

## 9. متطلبات الاستقلال عن الاستيراد

يجب أن يكون 100% من مكوّنات حزمة العتاد والبرمجيات:
- مُدرَجة في السجل الموحَّد للبرمجيات الروسية (قرار الحكومة رقم 1236)، أو
- معتمدة من FSTEC الروسي / FSB الروسي، أو
- تطويراً ذاتياً للمشغِّل بشيفرة مفتوحة المصدر وتجميع على أراضي الاتحاد الروسي.
