# Концепция архитектуры ЦП РСФСР
## Цифровая Платформа «Российский Сервис Финансирования Социального Развития»

**Автор и правообладатель концепции:** Кагиров Абдул-Хаким Ахмадович.
**Свидетельство о депонировании:** №4011265 от 19.12.2024 (ireg.pro), хеш ГОСТ-Р 34.11-2012.

## 1. Назначение и область применения

Платформа ЦП РСФСР — государственно значимая операторская информационная система выпуска цифровых финансовых активов (ЦФА) под залог имущества граждан и организаций Российской Федерации. В pilot-light режиме 2026–2028 целевое фондирование национальных и инфраструктурных проектов под обеспечение пула ЦФА организуется через инфраструктурный контур ВЭБ.РФ (Фабрика проектного финансирования, постановление Правительства РФ от 15.02.2018 №158) и Федеральное казначейство в рамках действующего БК РФ. Банк России — общий надзорный регулятор финансового рынка по ст. 4 86-ФЗ. На горизонте 2029+ возможен переход к модели целевой обеспеченной эмиссии Банка России (с применением цифрового рубля) при принятии соответствующего пакета НПА и положительном track-record pilot-light.

Документ описывает:
- общие архитектурные принципы платформы;
- логическую и физическую архитектуру;
- технологический стек на полностью импортозамещённых компонентах;
- модель надёжности и отказоустойчивости;
- требования к классификации информационной системы как объекта КИИ.

## 3. Логическая архитектура — 7 слоёв

1. **Канал собственника актива** — веб-портал, мобильные приложения iOS/Android/Windows, чат-бот в государственных каналах.
2. **Backend-for-Frontend (BFF)** — оптимизированный слой агрегации для каждого канала.
3. **API Gateway** — единая точка входа, аутентификация, rate-limiting, антифрод-фильтры.
4. **Бизнес-сервисы** — микросервисы: KYC, оценка, страхование, эмиссия ЦФА, смарт-контракты, реестр, эскроу-адаптер, отчётность, нотификации.
5. **Слой данных** — реляционные СУБД (Postgres Pro / Ред БД), хранилища документов, in-memory кеш (отечественный аналог Redis), объектное S3-совместимое хранилище.
6. **Слой интеграций** — адаптеры к ГИС (Госуслуги, ЕГРН, СМЭВ 3, ГИС ГМП, ФК, ЦБ, Росфинмониторинг, ФНС, НСПК).
7. **Слой безопасности** — DLP, SIEM, EDR, ГосСОПКА-агент, HSM-модули (КриптоПро HSM), системы мониторинга и аудита.

## 4. Физическая архитектура

### 4.1. География
- Минимум 3 действующих ЦОД на территории Российской Федерации в разных федеральных округах (Центральный, Сибирский, Уральский — пример).
- Топология active-active-active с синхронной репликацией для критичных контуров.
- Резервный «холодный» ЦОД для DR-сценариев.

### 4.2. Сетевая модель
- Двухуровневая защита периметра: 1-й уровень — государственная сеть «РТКомм», 2-й — собственный периметр с отечественными NGFW.
- Изолированный сегмент для интеграций с Банком России (СПФС, защищённый канал ВипНет).
- Полная сегментация по микросервисам (zero-trust network).

### 4.3. Платформа исполнения
- Контейнеризация на отечественных Kubernetes-дистрибутивах (Deckhouse Stronghold / Yandex Managed K8s on-prem-аналоги).
- Service Mesh с сертификатами КриптоПро для mTLS.

## 5. Технологический стек

### 5.1. Системное ПО
- Операционные системы: Astra Linux Special Edition, РЕД ОС, Альт Сервер — сертифицированы ФСТЭК.
- Гипервизор: zVirt / СБерCloud / vStack (отечественные аналоги VMware).
- Контейнеризация: отечественные дистрибутивы Kubernetes.

### 5.2. Базы данных и хранилища
- Реляционные: Postgres Pro Enterprise, Tantor Special Edition, Ред БД.
- Аналитические: Pangolin, Arenadata DB.
- Документоориентированные: Postgres Pro Multimaster JSONB; или отечественный аналог MongoDB (Atlasync).
- Объектное: Yandex Object Storage on-prem, Аркуда, S3-совместимый хост.

### 5.3. Брокеры и потоковая обработка
- Apache Kafka в дистрибутивах Arenadata Streaming / Yandex Managed Service for Apache Kafka.
- RabbitMQ как fallback для классических очередей.

### 5.4. DLT-слой (распределённый реестр ЦФА)
- На основе отечественной платформы класса «Мастерчейн» / Waves Enterprise; конкретный вендор согласуется с Банком России на этапе пилота.
- Хешировние операций — ГОСТ Р 34.11-2012.
- Электронная подпись операций реестра — ГОСТ Р 34.10-2012.

### 5.5. Криптография
- HSM: КриптоПро HSM / ВипНет HSM.
- SDK: КриптоПро CSP 5.0, ВипНет CSP.
- Сертификаты: Квалифицированные сертификаты УЦ, аккредитованных Минцифры.

### 5.6. Информационная безопасность
- SIEM: KOMRAD, MaxPatrol SIEM, R-Vision SIEM.
- EDR: Kaspersky EDR / R-Vision EDR.
- DLP: Solar Dozor / InfoWatch.
- WAF: PT Application Firewall / SolidWall.
- Системы анализа кода: PT Application Inspector, AppSec.Hub.
- VPN: ВипНет, КонтинентАП.

### 5.7. Наблюдаемость (Observability)
- Метрики: VictoriaMetrics, отечественный Prometheus-стек.
- Логи: Loki / отечественный аналог.
- Трассировка: Jaeger / отечественный OpenTelemetry-стек.
- Дашборды: Grafana OSS на собственной инфраструктуре.

## 6. Архитектурные принципы

1. **Государственная значимость на первом месте.** Безопасность, надёжность и регуляторная совместимость приоритетнее скорости разработки.
2. **Импортонезависимость 100%.** Все компоненты — из реестра Минцифры или сертифицированы ФСТЭК.
3. **Event-driven.** Все бизнес-процессы (выпуск ЦФА, мена, страховой случай) — асинхронные события с прослеживаемостью.
4. **Immutable audit log.** Все ключевые операции пишутся в DLT-реестр и в централизованный журнал аудита; данные неизменяемы.
5. **Zero-Trust.** Каждое обращение между микросервисами — с аутентификацией mTLS.
6. **Secure by Design.** Безопасность встроена с этапа архитектуры (Privacy by Design, KIIS by Design).
7. **Cloud-agnostic on-prem.** Платформа разворачивается на собственной инфраструктуре, без зависимости от иностранных облаков.
8. **API-first.** Все интеграции — через документированные API (OpenAPI 3.1).
9. **GitOps + IaC.** Инфраструктура и конфигурации — в git с автоматическим деплоем (Argo CD / отечественный аналог).
10. **Резервирование уровня N+2.** Критичные сервисы — с двойным резервированием.
